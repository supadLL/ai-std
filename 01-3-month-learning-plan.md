> 适用人群：C/C++ 后端/机器人/ROS 项目背景在职，只有晚上 + 周末时间，想转 AI Agent 方向
> 预期产出：2 个精品项目 + 面试准备
> 总时间预算：约 330 小时（3 个月 × 110 小时/月）
> 说明：这里会保留机器人/ROS 背景带来的工程优势，但不默认你熟悉 ROS 专业术语；后续解释优先用 C/C++、Linux 服务端、HTTP、状态机、缓存这些更通用的概念。


## 二、每个时间段的正确用法

```
┌─────────────────────────────────────────────────────┐
│  工作日晚上（2 小时）                                  │
│  → 看文档、读源码、写面试题答案、review 自己周末写的代码    │
│  → 不做的事：写新功能、搭环境、debug 复杂问题             │
│  → 原因：脑子已经累了，不适合深度编码，适合"阅读理解"       │
├─────────────────────────────────────────────────────┤
│  周六（8 小时）                                       │
│  → 主力编码日：写新功能、调参、跑实验                     │
│  → 上午 3h：写核心逻辑                                │
│  → 下午 3h：补边缘 case + 异常处理                     │
│  → 晚上 2h：整理文档 + 记录当天踩的坑                   │
├─────────────────────────────────────────────────────┤
│  周日（上午 4h + 下午 4h）                             │
│  → 上午：继续编码 / 性能优化                           │
│  → 下午：写项目 README + 更新简历 + 整理这周产出         │
│  → 晚上：休息，别碰代码                                │
└─────────────────────────────────────────────────────┘
```

**核心原则**：工作日不写新代码，周末不看书。

## 三、第 1 个月：RAG PDF 问答机器人（~110 小时）

> **C/C++/机器人项目背景学 Python 的心理建设**：Python 不是你转 AI 的障碍，是你最高效的工具。语法比 C++ 简单太多——没有指针、没有内存管理、没有头文件、没有模板元编程、没有 makefile。不用手动 `malloc/free`，不用跟 segmentation fault 搏斗。你过去做机器人项目积累的是工程直觉：模块拆分、异常处理、状态判断、实时性/稳定性意识。这些比会背 ROS 术语更重要。两周就能写项目，剩下的在项目里边写边学。

| | 做什么 | 时间 |
|---|---|---|
| 周一-周五晚 | 选定方向（建议智能客服）、注册 DeepSeek API、看 FastAPI 官方 Tutorial | 每晚 1.5h |
| 周六 | Docker 装 Qdrant、搭 FastAPI 项目骨架、跑通 `curl` 调通 DeepSeek | 8h |
| 周日 | 写第一个 API `/chat`，加超时控制 + 异常处理 + token 计数 | 6h |

**第 1 周产出**：一个 FastAPI 服务，`POST /chat` 能返回 LLM 回复，有超时控制。

**📖 第 1 周术语解释**：

| 术语 | 解释 | 类比 C/C++ / 通用工程经验 |
|------|------|---------------|
| **FastAPI** | Python 的 Web 框架，写 REST API 用的。性能和 Go 差不多。选它是因为 AI 生态全部在 Python 这边，用 C 从头撸 socket + HTTP 解析太慢了 | ≈ 就像你 C 里定义一个函数指针回调——声明路径 → 绑定处理函数 → 框架帮你处理 HTTP 协议细节；不需要理解 ROS service/action，也能理解它 |
| **DeepSeek** | 国产大模型厂商，API 兼容 OpenAI 格式，中文能力很强，价格只有 GPT-4 的 1/20。国内直接就能用，不需要翻墙 | ≈ 一个便宜好用的"AI 大脑"，你给它输入，它给你输出 |
| **DeepSeek API** | 通过 HTTP 请求调用 DeepSeek 大模型的服务接口。你发一段文字过去，它返回一段回答 | ≈ 本质就是一次 HTTP POST 请求，类似你用 libcurl 调第三方 REST 接口——发 JSON，收 JSON |
| **Qdrant** | 向量数据库。把你的文档转成数学向量存起来，然后可以做"语义搜索"——用户问"怎么退款"，不用写死关键词，它能理解意思去匹配 | ≈ 就像把数据存成高维数组，查询时不是精确匹配，而是算余弦距离找最近的——类似你在 C 里手写 kNN 搜索的工程版 |
| **Docker** | 容器化工具，一键启动各种中间件（不需要手动编译安装，一行 docker run 就起来了） | ≈ 像把程序、依赖库、运行环境打成一个包，到哪都能直接跑；如果你做过机器人部署，就可以理解成“把环境差异尽量抹平” |
| **LLM** | Large Language Model，大语言模型。DeepSeek、GPT-4、通义千问都是 LLM。本质：输入文字 → 输出文字 | ≈ 一个超级智能的文本处理函数——`std::string llm(const std::string& prompt)` |
| **Token** | LLM 计费的基本单位。一个中文字 ≈ 2 个 token，一个英文单词 ≈ 1.3 个 token。API 按 token 数量收费 | ≈ 就像串口通信按字节计费，token 就是 LLM 的"字节" |
| **curl** | 命令行 HTTP 客户端，终端里直接发请求测试接口 | ≈ 命令行版的 HTTP 调试器，测接口不用写代码，一行命令搞定 |

### 第 2 周：文档解析 + 切分

| | 做什么 | 时间 |
|---|---|---|
| 周一-周三晚 | 读 PyPDF2/pdfplumber 文档，搞清楚 PDF 解析的边界 case（扫描件、表格、图片） | 每晚 1.5h |
| 周四-周五晚 | 读 LangChain 的 text splitter 源码（就看 `RecursiveCharacterTextSplitter` 这一个文件） | 每晚 1.5h |
| 周六 | 写文档切分模块，实现 chunk_size 可配、overlap 可配 | 8h |
| 周日 | 跑 20 个真实 PDF 测试，记录不同 chunk_size 的切分效果，写下来 | 6h |

**第 2 周产出**：文档切分模块，3 种 chunk_size 的对比数据。

**📖 第 2 周术语解释**：

| 术语 | 解释 |
|------|------|
| **PyPDF2 / pdfplumber** | Python 读 PDF 文件的库。PyPDF2 适合纯文字 PDF，pdfplumber 能处理表格和复杂排版。扫描件 PDF 需要 OCR（光学字符识别） |
| **LangChain** | 目前最主流的 AI 应用开发框架。把"文档切分 → 向量存储 → LLM 调用"这些环节封装成标准模块，不用从零写。**注意**：我们只读它的源码学设计思路，不依赖它——面试时"我看过 LangChain 源码，但自己实现的切分逻辑"比"我用 LangChain 跑的"高级得多 |
| **Text Splitter** | 文档切分器。一份 50 页的 PDF，LLM 一次读取有长度限制（DeepSeek 约 128K token），而且内容太长检索精度会下降。所以要切成小块 |
| **RecursiveCharacterTextSplitter** | LangChain 内置的一种切分策略。按"段落 → 换行 → 句号 → 空格"的优先级逐级切分，尽量保持语义完整性。面试能讲清楚这一种就够了 |
| **chunk_size** | 每块文档的大小（按字符数 or token 数）。设太小（比如 100）语义不完整，设太大（比如 5000）检索不精准。常见范围 300-1000，需要根据你的文档类型调参 |
| **overlap** | 相邻两块的重复部分。比如 chunk_size=500, overlap=50，那第 1 块是前 500 字，第 2 块从第 451 字开始。防止关键信息刚好卡在两块边界被切断 |

### 第 3 周：向量存储 + 检索

| | 做什么 | 时间 |
|---|---|---|
| 周一-周三晚 | 读 Qdrant 官方文档（就看 search、upsert、collections 三块） | 每晚 1.5h |
| 周四-周五晚 | 了解 embedding 模型怎么选，对比 bge-large-zh / text2vec 的效果和价格 | 每晚 1.5h |
| 周六 | 写向量化 + 存储 + 检索的完整链路 | 8h |
| 周日 | 用 20 个问题测检索命中率，调 top_k | 6h |

**第 3 周产出**：RAG 检索链路跑通，有命中率数据。

**📖 第 3 周术语解释**：

| 术语 | 解释 |
|------|------|
| **Embedding（向量化）** | 把一段文字转成一个数学向量（比如 1024 个浮点数组成的数组）。意思相近的文字，向量也相近。比如"退款流程"和"退货步骤"的向量距离很近，所以能实现语义搜索 |
| **Embedding 模型** | 专门做"文字→向量"转换的模型。比通用 LLM 小、快、便宜。选对 embedding 模型对检索命中率影响巨大 |
| **bge-large-zh** | BAAI（北京智源）开源的**中文** embedding 模型，中文语义理解能力强，是目前中文 RAG 最常用的选择之一 |
| **text2vec** | 另一个开源中文 embedding 模型，比 bge 轻量一些，适合本地部署资源有限的场景 |
| **upsert** | Qdrant 的写入操作 = update + insert。如果向量已存在（相同 ID）就更新，不存在就插入。跟数据库的 upsert 一模一样 |
| **collections** | Qdrant 里的"表"。每个 collection 存一类文档的向量，可以独立设置索引参数。≈ 数据库的 table |
| **top_k** | 检索返回的最相关文档数量。top_k=5 就是取最相关的 5 个文档块喂给 LLM。太小可能漏信息，太大会引入噪音。这就是"调参"的核心对象之一 |
| **检索命中率** | 你自己测：准备 20 个问题，每个问题提前标好答案在文档哪一页。跑检索后看 top_k 结果里有没有命中正确位置。命中数/总数 = 命中率。**简历上的数字就从这来** |

### 第 4 周：RAG 回答 + 性能优化 + 文档

| | 做什么 | 时间 |
|---|---|---|
| 周一-周二晚 | 设计 RAG Prompt 模板，对比"有上下文"和"无上下文"的回答质量差异 | 每晚 1.5h |
| 周三-周四晚 | 写上所有异常处理链路（文档解析失败、检索超时、LLM 返回异常） | 每晚 1.5h |
| 周五晚 | 压测（用 locust），记录 P50/P99 延迟 | 2h |
| 周六 | 调整 chunk_size，重新跑检索命中率测试，记录优化前后的数据对比 | 8h |
| 周日 | **写项目 README**（架构图 + 技术选型理由 + 优化过程 + 性能数据），这是你面试的提词器 | 6h |

**📖 第 4 周术语解释**：

| 术语 | 解释 |
|------|------|
| **Prompt** | 发给 LLM 的输入文本。RAG 的 Prompt 通常包含：系统角色设定 + 检索到的文档上下文 + 用户问题 + 输出格式要求。Prompt 写得好不好，决定回答质量的上限 |
| **locust** | Python 压测工具。写一段 Python 脚本定义用户行为，然后模拟 N 个并发用户打你的接口。能看到 QPS、P50/P99 延迟。**这就是你的性能数据来源** |
| **P50 / P99 延迟** | P50 = 50% 的请求在这个时间内完成（中位数）；P99 = 99% 的请求在这个时间内完成（长尾）。简历写 P99 更有说服力，因为它反映了系统在最差情况下的表现 |
| **QPS** | Queries Per Second，每秒查询数。衡量接口吞吐量。作为写服务端程序的人你应该很熟 |
| **README** | 项目的门面文档，GitHub 仓库首页。面试官收到你的简历，大概率先打开你的 GitHub 看 README。写清楚：项目做什么、怎么跑、技术选型理由、性能数据 |

**第 1 个月产出一句话总结**：
> 基于 FastAPI + DeepSeek + Qdrant 的 PDF 智能问答系统，chunk_size 调优后检索命中率从 65% 提升至 92%，P99 延迟从 3.2s 降至 600ms。

## 四、第 2 个月：智能客服 Agent（~110 小时）

### 第 5 周：Function Calling 基础

| | 做什么 | 时间 |
|---|---|---|
| 周一-周四晚 | 读 DeepSeek / OpenAI 的 Function Calling 文档，搞清楚 tool 的 JSON Schema 怎么写 | 每晚 1.5h |
| 周五晚 | 设计 3-4 个工具的 schema（search_kb、query_order、create_ticket、transfer_human） | 2h |
| 周六 | 写 Agent 核心循环（LLM decide → tool execute → result → LLM decide again → reply） | 8h |
| 周日 | 写工具调用的超时控制 + 异常处理（每个工具超时不同怎么办？） | 6h |

**第 5 周产出**：一个能调用工具的 Agent，单轮对话能正确选工具并返回结果。

**📖 第 5 周术语解释**：

| 术语 | 解释 |
|------|------|
| **Function Calling** | LLM 不仅能输出文字，还能输出"我要调哪个函数，参数是什么"。你用 JSON Schema 定义工具，LLM 返回一个标准的 function call 请求。**这是 Agent 能"动手做事"的核心机制** |
| **JSON Schema** | 用 JSON 格式描述"这个工具叫什么、有什么参数、参数是什么类型"。比如 `{"name":"query_order","parameters":{"order_id":{"type":"string","description":"订单号"}}}`。LLM 看到这个就会自动从用户话里提取参数 |
| **Agent 核心循环** | Agent 的运行逻辑是一个 while 循环：1) 用户说一句话 2) LLM 决定要不要调工具 3) 如果要，执行工具拿到结果 4) 把结果再喂给 LLM 5) LLM 决定继续调工具还是直接回复用户。循环直到 LLM 认为可以回复了 |
| **search_kb** | 搜索知识库工具。用户问"怎么退货"，Agent 调这个工具查知识库里的退货政策 |
| **query_order** | 查询订单工具。用户问"我订单到哪了"，Agent 调用这个并传入订单号 |
| **create_ticket** | 创建工单工具。问题解决不了时，Agent 帮用户建一个客服工单 |
| **transfer_human** | 转人工工具。Agent 判断自己处理不了，把对话转给真人客服 |

### 第 6 周：多轮对话 + 记忆管理

| | 做什么 | 时间 |
|---|---|---|
| 周一-周三晚 | 看 LangChain Memory 模块的设计思路（不一定要用，但理解设计模式） | 每晚 1.5h |
| 周四-周五晚 | 设计记忆方案：短期记忆放 Redis，长期记忆放 SQLite | 每晚 1.5h |
| 周六 | 写短期记忆（滑动窗口 + 摘要压缩）+ 长期记忆（用户画像存储） | 8h |
| 周日 | 写多轮对话测试用例，测 10 组多轮对话，记录记忆是否连贯 | 6h |

**第 6 周产出**：有记忆的客服 Agent，多轮对话不会"失忆"。

**📖 第 6 周术语解释**：

| 术语 | 解释 |
|------|------|
| **多轮对话** | 不是一问一答，而是连续对话。用户说"我订单有问题"→ Agent 问"哪个订单"→ 用户说"上周那个"→ Agent 需要记住上下文才能理解"那个"指的是什么 |
| **LangChain Memory** | LangChain 框架的记忆管理模块。它把记忆抽象成标准接口：存、取、清、压缩。我们不一定要用它，但它的设计思路值得学 |
| **短期记忆** | 当前会话的上下文，存在 Redis 里，会话结束一段时间后自动删。存什么：最近 N 轮的对话记录 + 工具调用的中间结果。TTL（过期时间）一般设 30 分钟 |
| **长期记忆** | 跨会话的用户信息，存在数据库里。存什么：用户画像（VIP/普通）、历史工单、偏好。下次用户再来，Agent 能"认出"他 |
| **滑动窗口** | 短期记忆的淘汰策略。只保留最近 N 轮对话，旧的自动丢弃。比如 N=10，第 11 轮对话进来，第 1 轮就被丢弃。跟限流算法的滑动窗口一个概念 |
| **摘要压缩** | 聊天太长怎么办？用 LLM 把前 10 轮对话压缩成一段摘要（"用户反馈订单延迟，已查物流，预计明天到达"），节省 token |
| **Redis** | 高性能内存 key-value 存储。类似 STL 的 `std::unordered_map` 但数据在独立进程里、支持网络访问、可以设过期时间。适合存会话级的短期记忆 |
| **SQLite** | 轻量级嵌入式关系型数据库。一个 `.db` 文件就是数据库，不需要额外安装服务端。像把 SQL 直接跑在文件上，适合个人项目存长期记忆。C/C++ 可以直接通过 `sqlite3.h` 头文件调用 |

### 第 7 周：异常场景全覆盖

这一周是你面试时区别于"调 API 工程师"的关键。

| | 做什么 | 时间 |
|---|---|---|
| 周一-周二晚 | 梳理所有异常场景清单（至少列 15 个） | 每晚 2h |
| 周三-周五晚 | 逐一实现降级方案 | 每晚 2h |
| 周六 | 写熔断器（工具连续失败 N 次自动熔断）、写降级回复文案 | 8h |
| 周日 | 全部异常场景的集成测试 | 6h |

**异常场景清单**（面试能聊 20 分钟的东西）：

```
工具侧：
1. 工具 A 响应超时（3s）→ 降级：告知用户"正在查询，请稍候" + 后台异步重试
2. 工具 B 返回 500 → 降级：跳过该工具，用其他可用信息回复
3. 工具 C 连续失败 3 次 → 熔断 30 秒
4. 所有工具都不可用 → 降级：直接转人工

LLM 侧：
5. API rate limit → 排队重试，告知用户等待
6. API 返回异常 → 重试 1 次 + 降级为预设回复模板
7. 返回内容包含幻觉（检查引用来源）→ 标记为"不确定"并建议转人工

会话侧：
8. 用户输入为空 → 引导输入
9. 用户连续 3 轮不满意 → 主动转人工
10. 会话超时 → 清理短期记忆，保留长期记忆
```

**第 7 周产出**：一个线上不会崩的 Agent 系统（即使功能不多，但稳）。

**📖 第 7 周术语解释**：

| 术语 | 解释 |
|------|------|
| **熔断器（Circuit Breaker）** | 一个工具连续失败 N 次，直接标记为"不可用"，后续请求不再调它，直接走降级逻辑。过一段时间（比如 30 秒）放一次请求试一下，恢复了就关熔断。**本质就是一个状态机：CLOSED → OPEN → HALF_OPEN → CLOSED，跟你 C 里用 enum + switch 写的状态机一模一样** |
| **降级（Degradation）** | 某个环节失败了，不整体报错，而是用一个"次优方案"扛住。比如知识库检索超时 → 降级为用 LLM 自身知识回答 + 标注"建议稍后重试" |
| **Rate Limit** | LLM API 的调用频率限制。比如 DeepSeek 免费版可能每分钟最多 60 次请求。超了就返回 429 状态码。处理方式：捕获 429 → 等几秒重试 → 还不行就排队或拒绝 |
| **幻觉（Hallucination）** | LLM 编造不存在的事实，而且说得一本正经。比如问"退货政策"，知识库里没有，LLM 就现编一个。RAG 就是为了减少幻觉——让 LLM 只能根据检索到的文档回答。**这是面试必考题** |
| **异步重试** | 工具调用超时了，不阻塞用户，先返回一个"处理中"的提示，后台异步重试，好了再推送结果。跟消息队列的异步处理思路完全一致 |

### 第 8 周：写文档 + 成本优化

|---|---|---|
| 周一-周三晚 | 实现 Token 消耗统计 + 结果缓存（相同/相似问题不重复调 LLM） | 每晚 2h |
| 周四-周五晚 | 项目 2 的 README + 架构图 | 每晚 2h |
| 周六 | 简历第一版（两页，重点放 AI 项目，C/C++/机器人项目工程经验当背景和加分项） | 8h |
| 周日 | 休息一天，准备进入第 3 个月 | - |

**📖 第 8 周术语解释**：

| 术语 | 解释 |
|------|------|
| **Token 消耗统计** | 每次调 LLM API 记录消耗的 token 数，按天/周汇总。一来方便算成本，二来如果 token 消耗突然飙升说明可能有问题（比如有人在刷接口） |
| **结果缓存** | 相同/相似的问题不重复调 LLM。用户 A 问"退货流程"，用户 B 也问"退货流程"→ 直接从 Redis 返回缓存结果。**本质上就是一个带过期时间的 key-value cache，跟你在 C++ 里用的 unordered_map 思路一样，只是多了 TTL** |
| **模型分级** | 简单问题用小模型（如 DeepSeek-V3），复杂问题用大模型（如 DeepSeek-R1）。小模型便宜 10 倍，大模型能力强。通过意图识别自动路由 |

**第 2 个月产出一句话总结**：
> 基于 Function Calling 的智能客服 Agent，支持多轮对话记忆管理、工具调用超时熔断降级、LLM Token 成本优化。
